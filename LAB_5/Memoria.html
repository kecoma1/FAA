1. Detalles de la implementación¶

Lo primero sera mencionar la estructura que tiene cada individuo. Para crear esta estructura se ha hecho una clase interna en la cual se define a un individuo con los siguientes datos.

class Individuo:
    fitnessValue # Valor de fitness del individuo
    priori # Clase mayoritaria de los datos
    reglas = [
        {"regla": [int], "conclusion": int} # Cada regla es un diccionario
        ...
    ]

Para representar las reglas se usa un diccionario, en este diccionario tenemos la clave "regla" en la cual el valor es una lista de enteros. Cada entero de esta lista hace referencia a la cadena de bits para un atributo, por ejemplo: Si tenemos 3 atributos A, B y C, tendremos 3 enteros en la lista, si cada atributo tiene 3 valores, se usaran 3 bits de cada entero. Para operar con estos bits se usan bit operators (&, |).

La conclusin es un entero que marca 0 o 1.

1.1 Generación de la población inicial¶

Las reglas se crean de forma aleatoria, asignando para cada atributo una cadena aleatoria (de la cual solo se van a usar los N primeros bits, siendo N el nmero de posibles valores en el atributo). Para este anlisis se usaran individuos con un mximo de 5 reglas.

1.4 Clasificación¶

La clasificacin es simple, se coge al mejor individuo del entrenamiento y se aplican las reglas, pero, cmo se aplican estas reglas? Al utilizar cadenas de bits para cada atributo es simple comparar si un valor se cumple en una regla o no, por ejemplo: Si tenemos el atributo A con los posibles datos 'a', 'b', 'c', tendramos una cadena de bits para este atributo con 3 bits. Si recibimos el valor 'a', recibimos la cadena 001 (un entero con valor 1), si recibimos 'b' 010 (entero con valor 2) y si recibimos 'c' 100 (entero con valor 4). Con puertas lgicas es muy simple comparar estos valores. Si tenemos la regla 101 (entero con valor 5) y recibimos el valor 001 (a) (aplicamos la puerta lgica and sobre los valores enteros 5 y 1, 101 & 001), pues esa parte de la regla se aprobara para este atributo y hara falta hacerlo con todos los dems, si se cumplen todos devolvemos la conclusin. Si varias reglas se cumplen, escogemos la conclusin mayoritaria, y si no se cumple ninguna regla, devolvemos la clase mayoritaria (a priori).

En el entrenamiento (tambin en la clasificacin) puede pasar que varias relgas se cumplan o que no se cumpla ninguna, en el primer caso lo que se hace es devolver la conclusin mayoritaria, en el segundo caso se devulve la clase a priori (clase mayoritaria).

3. Análisis¶

Lo primero que destaca a la vista es el pronunciado error. Una de las posibles razones es por la validacin en si, al no entrenar con todos los datos puede que las reglas se sobreajusten a los datos (el fitness del mejor individuo suele rondar el 70%-90% de aciertos, puede que se sobreajuste a los datos de entrenamiento), puede que el mejor individuo tras el entrenamiento tenga reglas que no se adecen correctamente a los datos de test, esto provoca que se recurra mucho al priori provocando quizs mas errores de los deseados. Otra razn puede ser el nmero de reglas, este puede ser un factor clave en relacin al error obtenido. Por ltimo, otra razn que se me ocurre es por el hecho de que el algoritmo en si es muy aleatorio, puede que esa aleatoriedad no permita una clasificacin mejor.

Algo que se puede ver, es que en algunos individuos se repiten reglas, asumo que esto significa que estas reglas son buenas, ya que al cruzar individuos se han mantenido durante generaciones. Pero esto tambin tiene otra explicacin, y es el propio mtodo de cruce, el cruce inter reglas cruza las reglas de dos individuos sin modificarlas, esto puede causar lo anteriormente mencionado. En el cruce intra reglas no vemos tanta repeticin de reglas, ya que las reglas son modificadas.

Otro factor relevante es la probabilidad de mutacin. Una probabilidad mayor podra acelerar el proceso de encontrar un muy buen individuo, pero hay que tener cuidado, una probabilidad muy grande puede provocar ms errores, es muy similar al descenso de gradiente en regresin logstica, si la constante de aprendizaje era muy alta, puede que no encontrasemos nunca mnimos, lo mismo pasa aqu, una probabilidad muy alta de mutacin puede que lastre al algoritmo. El mtodo de mutacin tambin es importante, ya que en la mutacin estandar cambiamos un nico bit de una nica regla, mientras que la mutacin de reglas pasa por todos los bits tratando de cambiarlos. Podemos decir que en uno se tira un dado, y en otro se tira el dado muchas veces.

Otra cosa que se ve de los mejores individuos es el nmero de reglas, a pesar de que la razn puede decirnos que "cuantas ms mejor", vemos que los mejores individuos no siempre tienen el nemro mximo de reglas. En muchos casos tienen 2 o 3, aunque de media dira que hay 3 reglas.

En cuanto al tamao de la poblacin, este es importante. Un tamao de mutacin amplio permite variedad de individuos aunque, no es necesario poner de ms. Hay que tener en cuenta el nmero de atributos y el nmero de posibles valores de cada uno, ya que si ponemos ms individuos que combinaciones no tendra sentido, aunque tambin es cierto que el nmero de combinaciones asciende rpidamente si el dataset es grande, haciendo casi imposible crear muchos individuos.

Por ltimo, el nmero de generaciones. Cuantas ms generaciones mejor, aunque hay que poner un lmite, ya que en un punto ser muy difcil para el algoritmo encontrar un individuo mejor que el que hay y lo nico que haremos ser gastar recursos (esto se ve mejor en el siguiente apartado).
